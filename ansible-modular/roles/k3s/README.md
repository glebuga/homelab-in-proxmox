# Роль k3s

Роль разворачивает лёгкий Kubernetes-кластер (k3s) на группе хостов `kuber`
в динамическом inventory. Состоит из одной control-plane ноды (master) и
одной или нескольких worker-нод, которые к ней присоединяются.

## Топология

- **Первая** нода в группе `kuber` (либо любая, в имени которой есть
  `master`) запускает k3s **server** (control plane).
- Все остальные ноды `kuber` запускают k3s **agent** и присоединяются к мастеру.

В текущем окружении:
- `k3s-205` (10.0.0.205) — **master** (control-plane)
- `k3s-206` (10.0.0.206) — **worker**

Роль определяет, кто есть кто, сама через переменную `k3s_role`
(см. `defaults/main.yml`): первый хост группы `kuber` считается мастером.

## Как это работает (пошагово)

### 1. Определение роли
В начале проставляется факт `k3s_is_master` (`true` для мастера, `false`
для воркера). Дальше вся логика ветвится по этому флагу.

### 2. Мастер (`k3s_is_master == true`)
1. Проверяет, есть ли уже бинарь `/usr/local/bin/k3s` (`stat`). Если нет и
   не задан `k3s_skip_download` — качает нужную версию с GitHub releases.
   Проверка `stat` важна: в ограниченных сетях бинарь часто кладут вручную,
   и повторное скачивание может упасть по таймауту.
2. Запускает официальный инсталлятор `get.k3s.io` в режиме **server** с
   отключёнными компонентами из `k3s_disable_components` (по умолчанию
   `traefik servicelb`). Отключение traefik (ingress) и servicelb (LoadBalancer)
   убирает лишние процессы/поды, которые не нужны для простого кластера,
   который только тянет образы из registry (Harbor).
3. Ждёт появления файла `/var/lib/rancher/k3s/server/node-token` — это
   означает, что сервер поднялся и готов выдавать токен для воркеров.

### 3. Воркер (`k3s_is_master == false`)
1. **Читает node-token прямо с мастера** через `slurp` с `delegate_to:
   "{{ k3s_master_ip }}"`. Токен НЕ хардкодится в коде и НЕ передаётся между
   плеями playbook'а — воркер сам забирает его с мастера в момент запуска.
2. При необходимости докачивает бинарь (аналогично мастеру, через `stat`).
3. Запускает инсталлятор в режиме **agent**, передавая
   `K3S_URL=https://<IP мастера>:6443` и `K3S_TOKEN=<токен>`. Агент
   регистрируется в кластере и начинает принимать поды.

### 4. Верификация (только на мастере)
Блок `Verify cluster state (master)` выполняется **только при
`k3s_is_master == true`**, поэтому на воркере эти таски показываются как
`skipping`. Мастер выводит `kubectl get nodes -o wide` и
`kubectl get pods -A -o wide`, чтобы убедиться, что обе ноды в статусе
`Ready`, а системные поды запущены.

> Проверка живёт на мастере и выводится один раз.

## Идемпотентность

- `get_url` защищён проверкой `stat` — не качает бинарь заново, если он уже
  есть (не падает при таймаутах сети при повторных прогонах).
- Установка server/agent защищена `creates: /etc/systemd/system/k3s.service`
  (или `k3s-agent.service`) — повторный запуск playbook'а не переустанавливает
  уже стоящий сервис.
- `systemd` запускает сервисы с `daemon_reload: true`, чтобы systemd
  подхватил свеже созданные unit-файлы.

Повторный запуск `ansible-playbook playbooks/k3s.yml` проходит без изменений
(`changed=0`), если кластер уже развёрнут.

## Использование

```bash
cd ansible-modular
ansible-playbook -i inventory/terraform_inventory.py playbooks/k3s.yml
```

Порядок важен и задан в playbook: сначала `kuber[0]` (мастер), затем
`kuber[1:]` (воркеры). Мастер всегда устанавливается первым, иначе воркеру
некуда будет присоединяться.

## Настраиваемые параметры (`defaults/main.yml`)

| Переменная | По умолчанию | Назначение |
|------------|--------------|------------|
| `k3s_version` | `v1.36.2+k3s1` | Версия релиза k3s |
| `k3s_role` | авто: первый в `kuber` = master | Роль хоста (master/worker) |
| `k3s_master_ip` | `groups['kuber'][0]` | IP мастера, к которому цепляются воркеры |
| `k3s_api_port` | `6443` | Порт API-сервера |
| `k3s_disable_components` | `traefik servicelb` | Отключаемые встроенные компоненты |
| `k3s_extra_args` | `""` | Доп. аргументы серверу/агенту |
| `k3s_skip_download` | `false` | Не качать бинарь, использовать уже лежащий в `/usr/local/bin/k3s` |

## Заметки по безопасности

- **Токен мастера не хранится в коде.** Он читается с мастера в рантайме
  через `slurp` + `delegate_to` и существует только в памяти durante прогона.
- `kubectl` предоставляется самим k3s по пути `/usr/local/bin/kubectl` на
  мастере.
- Отключённые `traefik`/`servicelb` уменьшают поверхность атаки и нагрузку —
  для bare-кластера, который только тянет образы из registry, они не нужны.
